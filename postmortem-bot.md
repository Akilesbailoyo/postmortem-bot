# postmortem-bot

A Slack bot that auto-drafts incident post-mortems in under 5 minutes.

---

## How it works — plain language

This is the full flow from the moment an engineer types the command to the moment the draft appears. No jargon.

```
ENGINEER types: /postmortem #incident-polygon-0703
      │
      ▼
BOT extracts the channel name from the command
      │
      ▼
BOT fetches three things IN PARALLEL:
      │
      ├── [1] SLACK — reads every message in #incident-polygon-0703
      │         Why: the thread is the raw incident narrative.
      │         Who said what, when the rollback started, when things recovered.
      │         How: Slack conversations.history API (paginated — gets everything)
      │
      ├── [2] GITHUB — scans the Slack messages for any github.com/pull/ links,
      │         then fetches: PR title, who merged it, when, which files changed,
      │         and any issues referenced in the PR body (e.g. "Fixes #387")
      │         Why: most incidents correlate with a recent deployment.
      │         How: GitHub REST API /repos/{owner}/{repo}/pulls/{number}
      │
      └── [3] OPSGENIE — queries for alerts that fired during the incident window
                (derived from the timestamps of the first and last Slack messages)
                Returns: alert name, severity, when it triggered, when it resolved,
                which service it was tagged to.
                Why: the alert is the system's own record of what broke and when.
                How: OpsGenie /v2/alerts API with a time-range + service-tag query
      │
      ▼
BOT checks: do we have enough to generate a useful draft?
  → Slack thread empty?   → ask the engineer to confirm the channel name
  → GitHub data missing?  → proceed, note the gap in the draft
  → OpsGenie missing?     → proceed, note the gap in the draft
      │
      ▼
BOT calls Claude (claude-3-5-sonnet-20240620) with all collected data
and a structured prompt that says:
  - Use only the data provided, do not invent facts
  - Do not name individuals as causes — focus on systems and processes
  - If something cannot be determined, say so explicitly
  - Output exactly 7 sections: Summary, Timeline, Root Cause,
    Contributing Factors, Impact, Action Items, What Went Well
      │
      ▼
CLAUDE returns a structured draft
      │
      ▼
BOT posts the draft to the incident Slack channel with a clear label:
  "Draft generated by postmortem-bot. Engineer review required before publishing."
      │
      ▼
ENGINEER reviews, corrects anything wrong, and publishes.
The bot never publishes automatically. Human judgment is always the final step.
```

**The core insight:** most post-mortem time is spent gathering information that already exists — the Slack thread, the GitHub history, the alert timeline. The bot automates that gathering in seconds. The engineer adds what only they know: whether the root cause is correct and which action items are worth doing.

## Project structure

```
postmortem-bot/
├── .env.example          ← copy to .env and fill in your keys
├── .gitignore
├── package.json
├── tsconfig.json
├── README.md
└── src/
    ├── types.ts          ← shared data shapes used across all files
    ├── generate.ts       ← LLM call: takes context, returns post-mortem draft
    ├── demo.ts           ← standalone runner (mock or real mode, no Slack app needed)
    ├── index.ts          ← production Slack bot (/postmortem slash command handler)
    ├── fetchers/
    │   ├── slack.ts      ← reads the incident Slack thread (real Slack API)
    │   ├── github.ts     ← finds and fetches linked GitHub PRs (real GitHub API)
    │   └── opsgenie.ts   ← queries OpsGenie for alerts in the incident window (real API)
    └── mock/
        └── data.ts       ← hardcoded realistic incident data (used when MODE=mock)
```

## Setup

### 1. Mac setup from scratch (if you've never run code on this machine)

Open Terminal (Cmd+Space → type Terminal → Enter):

```bash
# Install Homebrew (Mac package manager)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# After it finishes, it prints two commands to add it to your PATH — run those exactly as printed.
# They look like:
#   (echo; echo 'eval "$(/opt/homebrew/bin/brew shellenv)"') >> ~/.zprofile
#   eval "$(/opt/homebrew/bin/brew shellenv)"

# Install Node.js (npm comes with it)
brew install node

# Verify
node --version    # should be v18 or higher
npm --version     # should be 9 or higher

# Install TypeScript runner
npm install -g ts-node typescript
```

### 2. Clone and install

```bash
git clone https://github.com/your-org/postmortem-bot.git
cd postmortem-bot
npm install
```

### 3. Configure environment

```bash
cp .env.example .env
# Open .env and fill in your keys (see .env.example for what each one is and where to get it)
```

## Running the demo (no Slack app required)

### Option A — Full mock mode (no API keys, no accounts, completely offline)

```bash
npm run demo:mock
```

Uses hardcoded incident data from `src/mock/data.ts`. The data shapes are identical to what the real APIs return, so switching to real mode requires no code changes. Set `ANTHROPIC_API_KEY` in `.env` to also get a live LLM-generated draft. Without it, the bot prints the context it would send and explains what would happen next.

### Option B — Real APIs, mock incident channel

```bash
# Set your keys in .env first, then:
npm run demo:real
# You'll be prompted for a real Slack channel ID
```

### Run online with no local install

For a zero-install demo, go to **onecompiler.com/nodejs** — no account needed.

1. Go to [onecompiler.com/nodejs](https://onecompiler.com/nodejs)
2. Delete the default code
3. Paste the contents of `src/mock/data.ts` first, then the contents of `src/demo.ts` below it (removing the import lines at the top of demo.ts since everything is in one file)
4. Hit Run

This runs the mock flow and prints the full simulated output. No keys, no accounts.

## Running the production bot

```bash
# Make sure all keys are set in .env
npm start
# Bot starts on port 3000

# For local development, expose the port with ngrok:
ngrok http 3000
# Then set the Request URL in your Slack app to: https://your-ngrok-url.io/slack/events
```

In your Slack app settings (api.slack.com/apps):
- Add a slash command: `/postmortem`
- Request URL: your server's public URL + `/slack/events`
- Required OAuth scopes: `channels:history`, `chat:write`, `commands`

## Common errors

| Error | Cause | Fix |
|---|---|---|
| `SLACK_BOT_TOKEN is not set` | Missing key in .env | Copy .env.example to .env and fill in the value |
| `Slack API error: channel_not_found` | Bot not in the channel | Invite the bot: `/invite @postmortem-bot` in the incident channel |
| `Slack API error: not_in_channel` | Same as above | Same fix |
| `GITHUB_TOKEN is not set` | Missing key in .env | Create a token at github.com → Settings → Developer settings → Personal access tokens |
| `OPSGENIE_API_KEY is not set` | Missing key in .env | OpsGenie → Settings → API Key Management |
| `Authentication error` from Anthropic | Wrong or expired API key | Check console.anthropic.com → API Keys |
| Draft cuts off mid-sentence | Response too long for token limit | Increase `max_tokens` in `src/generate.ts` from 1500 to 2000 |
| LLM invents facts not in the Slack thread | Edge case with short threads | The prompt says "do not invent" but add: "If a section cannot be determined, write 'Insufficient data — engineer to complete'" |
